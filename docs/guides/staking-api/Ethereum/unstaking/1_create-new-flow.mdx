title: Create New Unstaking Flow

To initiate the unstaking process, create a new flow with a `POST` request to `/api/v1/flows`.

### Parameters

- `flow` &mdash; `object`
  - `network_code` &mdash; `string` Network on which this flow operates (ex. `ethereum`) (required).
  - `chain_code` &mdash; `string` Chain on which this flow operates (ex. `goerli`) (required).
  - `operation` &mdash; `string` The operation to perform (ex. `unstaking`) (required).
  - `version` &mdash; `string` API version to use (ex. `v1`) (optional).

#### Sample cURL Request

```bash {5-10}
curl -X POST 'https://eth-slate.datahub.figment.io/api/v1/flows' \
-H 'Content-Type: application/json' \
-H 'Authorization: <api_key>' \
-d '{
  "flow": {
    "network_code": "ethereum",
    "chain_code": "goerli",
    "operation": "unstaking",
    "version": "v1"
  }
}'
```

### Response

- `id` &mdash; `string` ID of the newly created flow.
- `state` &mdash; `string` The current state of the flow, `initialized`.

The response indicates two possible actions:

- `broadcast_unstaking_msg` &mdash; use this action to broadcast a transaction to unstake your ETH.
- `unstaking_msg_expired` &mdash; use this action to indicate that the unstaking message has expired. This action has no inputs.

The `broadcast_unstaking_msg` action indicates three fields for data collection:

- `validator_index` &mdash; The validator from which the ETH will be unstaked.
- `epoch` &mdash; The Ethereum epoch number during which the ETH will be unstaked.
- `signature` &mdash; A BLS proof of possession, i.e. a proof that the private key corresponding to the `validator_index` is known by the signer.
